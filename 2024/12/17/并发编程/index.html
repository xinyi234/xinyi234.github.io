并发编程

发表于2024-12-17|更新于2024-12-25|CSAPP

基于进程的并发编程

服务器监听一个监听描述符的连接请求，假设服务器接受客户端1的连接请求并返回一个已连接描述符，服务器派生子进程，这个子进程获得服务器描述符表的完整副本。子进程关闭他的副本中的监听描述符3，父进程关闭已连接描述符4的副本。

父子进程之间共享文件表，但是不共享用户地址空间。为了共享信息，它们必须使用显式的IPC机制

Unix IPC通常是所有允许进程和同一台主机的其他进程进行通信的技术，包括管道，先进先出，系统V共享内存，系统V信号量等

基于IO多路复用的并发编程

使用select函数，要求内核挂起进程，只有再一个或多个IO事件发生后，才将控制返回给应用程序。select 函数处理类型为 fd_set 的集合，也叫做描述符集合。逻辑上，我们将描述符集合看成一个大小为 n 的位向量

select 函数有两个输入：一个称为读集合的描述符集合（fdset）和该读集合的基数（n）（实际上是任何描述符集合的最大基数）。

IO多路复用的事件驱动服务器运行在单一进程上下文中，数据共享很容易，但是很难处理部分文本行的问题，不能充分利用多核处理器。

基于线程的并发编程

线程是运行在进程上下文的逻辑流，每个线程有自己的线程上下文，包括一个唯一的整数线程ID,栈，栈指针，程序计数器，通用目的寄存器和条件码。所有运行在一个进程的线程共享该线程的整个虚拟地址空间。

主线程和其他线程的区别仅在于它总是进程中第一个运行的进程，一个进程可以杀死他的任何对等线程，或者等待他的任意对等线程终止。

Posix线程是C程序处理线程的一个标准接口

1.创建线程 pthread_create函数

当 pthread_create 返回时，参数 tid 包含新创建线程的 ID。新线程可以通过调用 pthread_self 函数来获得它自己的线程 ID。

2.终止线程 pthread_exit函数，它会等待所有其他对等线程终止，然后再终止主线程和整个进程，返回值为 thread_return。

3.终止当前进程 pthread_cancel函数

4.等待其他线程终止 pthread_join函数，只能等待一个指定的线程终止，pthread_join 函数会阻塞，直到线程 tid 终止，将线程例程返回的通用 (void*) 指针赋值为 thread_return 指向的位置，然后回收已终止线程占用的所有内存资源。

5.分离线程 pthread_detach函数，线程能够通过以 pthread_self() 参数的 pthread_detach 调用来分离它们自己。

读者写者模型

读者写者模型（Readers-Writers Problem）是操作系统中经典的同步问题之一，通常用来处理多线程并发访问共享数据时的冲突。该模型的核心是解决 多个读者线程 和 写者线程 对共享资源（如文件、数据库）的读写访问问题，同时确保数据一致性和线程安全性。

模型描述

  • 读者线程：只读取共享资源，不会修改内容。多个读者可以同时读取共享资源，不会互相干扰。
  • 写者线程：会修改共享资源，修改时可能破坏数据的一致性，因此需要独占资源。

目标

  1. 数据一致性：确保写者在修改数据时，不允许其他读者或写者访问数据。
  2. 读者并发性：允许多个读者同时读取资源，提高效率。
  3. 避免线程饥饿：读者或写者都不应因对方的过多优先而长时间等待。

两种常见问题

1. 第一类读者写者问题（读者优先）

  • 允许多个读者同时访问共享资源。
  • 写者必须等待，直到所有读者完成读取操作。
  • 问题：写者可能因为读者频繁读取而饥饿（得不到执行机会）。

2. 第二类读者写者问题（写者优先）

  • 当有写者等待时，不允许新的读者进入，即写者优先获得访问权。
  • 问题：可能导致读者饥饿，特别是当写者频繁到来时。

解决方案

使用信号量（Semaphore）或互斥锁（Mutex）来控制线程同步，同时通过计数器跟踪当前活动的读者数量。

  • **rw_mutex**：读写锁，控制对共享资源的访问。
  • **read_count**：记录当前读者的数量。

3. 使用读写锁（读写锁更加简化）

现代操作系统提供 读写锁（Reader-Writer Lock）作为抽象工具，简化了读者写者问题的实现。

  • 读模式：
    • 多个线程可以同时持有读锁。
  • 写模式：

总结

特点 读者优先 写者优先
读者并发性 高 低
写者饥饿 可能发生 不会发生
读者饥饿 不会发生 可能发生
适用场景 读操作频繁，写操作少 写操作重要或频繁，需优先考虑写者安全

生产者消费者模型

模型概述

  • 核心思想：
    • 生产者线程负责生成数据（或任务），将其放入共享缓冲区。
    • 消费者线程负责从共享缓冲区中取出数据进行处理。
    • 生产者和消费者之间通过共享缓冲区进行通信。
  • 目标：
    • 解决多线程协作的问题，确保生产者和消费者线程之间的数据流畅通，并避免竞争条件、死锁等问题。

关键问题与解决方案

  1. 数据同步：
    • 多个线程同时访问共享缓冲区时，需要避免数据竞争。
    • 解决方法：使用互斥锁（Mutex）或信号量（Semaphore）来控制对缓冲区的访问。
  2. 缓冲区状态：
    • 缓冲区可能满（生产者需要等待）或空（消费者需要等待）。
    • 解决方法
      • 使用信号量计数缓冲区的空闲单元数（empty）和已用单元数（full）。
      • 通过 P 和 V 操作协调生产者和消费者的行为。
  3. 线程间同步：
    • 保证生产者和消费者之间的执行顺序正确。
    • 解决方法
      • 使用条件变量（Condition Variable）来协调线程等待和唤醒。

实现步骤

  1. 定义一个共享缓冲区（如队列、数组）。
  2. 使用信号量和互斥锁来管理缓冲区访问。
  3. 编写生产者线程：
    • 等待缓冲区有空闲单元（信号量 empty）。
    • 加锁后向缓冲区写入数据。
    • 释放锁并增加缓冲区已用单元数（信号量 full）。
  4. 编写消费者线程：
    • 等待缓冲区有数据（信号量 full）。
    • 加锁后从缓冲区读取数据。
    • 释放锁并增加缓冲区空闲单元数（信号量 empty）。

预线程化

模型概述

  • 核心思想：
    • 在程序启动时，预先创建一组线程（线程池），线程在任务到达时执行相应的工作。
    • 避免动态创建线程带来的性能开销。
  • 目标：
    • 提高线程管理效率，减少频繁创建和销毁线程的开销。
    • 实现高效的任务调度。

优点

  1. 性能优化:
    • 线程池中的线程是预先创建的，避免了频繁创建和销毁线程所带来的开销。
  2. 资源复用：
    • 线程可以被多次复用，减少了系统资源的浪费。
  3. 易于管理：
    • 通过限制线程池大小，可以控制系统中活跃线程的数量，避免资源争抢。

实现步骤

  1. 初始化线程池
    • 创建固定数量的线程并将其存储在一个线程池中。
  2. 任务分发
    • 主线程接收任务，并将任务放入任务队列。
  3. 线程执行任务
    • 线程池中的空闲线程会从任务队列中取出任务进行处理。
    • 任务完成后，线程回归空闲状态，等待新的任务。

关键组件

  1. 任务队列：
    • 用于存储待处理的任务。
    • 通常使用线程安全的数据结构（如阻塞队列）。
  2. 线程池管理器：
    • 负责创建、销毁线程池中的线程。
    • 分配任务给线程。
  3. 工作线程：
    • 从任务队列中取任务并执行。

生产者-消费者模型 vs 预线程化模型

特点 生产者-消费者模型 预线程化模型
线程创建 动态，生产者和消费者可独立生成或销毁线程 静态，线程在启动时创建，复用线程执行任务
使用场景 多线程协作问题，如文件读写、数据处理 高并发任务处理，如 Web 服务器
性能 灵活性高，但线程创建销毁频繁可能影响性能 性能更高，适合长时间高负载任务场景

多线程程序中的共享变量

每个线程都有它自己独立的线程上下文，包括线程 ID、栈、栈指针、程序计数器、条件码和通用目的寄存器值。每个线程和其他线程一起共享进程上下文的剩余部分。这包括整个用户虚拟地址空间，它是由只读文本（代码）、读/写数据、堆以及所有的共享库代码和数据区域组成的。线程也共享相同的打开文件的集合。

寄存器是从不共享的，虚拟内存总是共享。

将变量映射到内存：全局变量，本地自动变量，本地静态变量。如果变量的一个实例被一个以上的线程引用，那么它是共享的

用信号量同步线程

进度图将 n 个并发线程的执行模型化为一条 n 维笛卡儿空间中的轨迹线。操作共享变量的指令构成了一个临界区，这个临界区不应该和其他线程的临界区交替执行。

信号量s是具有非负整数值的全局变量，只能进行P和V操作。P将s减1，V将s加1。P和V操作都是不可分割的

Posix标准定义了许多操作信号量的函数

二元信号量也被成为互斥锁，用P和V操作将临界区包围起来

信号量可以调度对共享资源的访问

1.生产者-消费者问题

互斥访问缓冲区并且调度对缓冲区的访问。如果缓冲区是满的（没有空的槽位），那么生产者必须等待直到有一个槽位变为可用。与之相似，如果缓冲区是空的（没有可取用的项目），那么消费者必须等待直到有一个项目变为可用。

2.读者-写者问题

第一类读者—写者问题，读者优先，要求不要让读者等待，除非已经把使用对象的权限赋予了一个写者。

第二类读者—写者问题，写者优先，要求一旦一个写者准备好可以写，它就会尽可能快地完成它的写操作。

线程安全

线程不安全函数：

  • 不保护共享变量的函数
  • 跨越多个调用的状态的函数
  • 返回指向静态变量的指针的函数
  • 调用线程不安全函数的函数
